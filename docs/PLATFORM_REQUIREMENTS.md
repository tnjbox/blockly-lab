## 三、平台總目標

本平台目標是建立一套適合國中小資訊科技與科技教育使用的 Blockly 教學平台，讓學生可以透過積木程式操作 ESP8266 SmartRingController，並逐步銜接：

1. 實體運算與硬體互動
2. Blockly 積木程式設計
3. JavaScript 程式碼觀察
4. 陣列與資料結構概念
5. 任務導向學習
6. 程式解題與競賽評分
7. 學習歷程與成績上傳

---

## 四、長期系統架構

長期架構目標如下：

```text
Scratch / OSEP Scratch 介面
Blockly / JavaScript 介面
教師測試頁
        ↓
共用 SmartRing WebSerial JSON 協定
        ↓
同一份 ESP8266 SmartRingController 統一韌體
        ↓
不用因為切換介面就重新燒錄
```

## 五、使用者角色

### 1. 學生

學生主要透過 Blockly 積木完成任務，並在學習過程中觀察 SmartRing 的實體回饋。

學生需要的功能：

- 讀取課程任務
- 使用 Blockly 拖拉積木
- 執行程式
- 控制 SmartRing LED
- 讀取 SmartRing 按鈕狀態
- 存檔與載入自己的積木作品
- 在學習模式中反覆測試
- 在競賽模式中提交成績

### 2. 教師

教師主要負責課程設計、任務測試、平台部署與學生學習成果檢核。

教師需要的功能：

- 建立或維護課程代碼
- 設計任務說明
- 測試 SmartRing 連線與指令
- 準備起始積木檔
- 檢查學生程式碼或積木檔
- 未來可透過 Google Sheet 收集學習結果

### 3. 進階學生 / 競賽學生

進階學生可以逐步從 Blockly 過渡到 JavaScript 與程式解題。

進階功能需求：

- 觀察 Blockly 產生的 JavaScript
- 完成陣列與邏輯任務
- 使用測資評分
- 進行限時或正式競賽模式練習

## 八、SmartRing WebSerial 需求

### 1. 連線需求

平台應透過 WebSerial 與 ESP8266 SmartRingController 通訊。

基本通訊設定：

| 項目 | 規格 |
|---|---|
| 通訊方式 | WebSerial |
| baudRate | 115200 |
| 資料格式 | 一行一筆 JSON |
| 換行格式 | `\n` |

### 2. ESP8266 傳回狀態格式

ESP8266 應持續輸出按鈕狀態：

```json
{"btn":[false,false,false,false,false,false,false,false],"func":0,"mode":0}
```

按鈕順序：

| index | 按鈕 | 方向 / 功能 |
|---:|---|---|
| 0 | BTN1 | F |
| 1 | BTN2 | B |
| 2 | BTN3 | L |
| 3 | BTN4 | R |
| 4 | BTN0 | U |
| 5 | BTN5 | D |
| 6 | BTN6 | O |
| 7 | BTN7 | C |

### 3. 前端送出 LED 指令格式

設定單顆 LED：

```json
{"cmd":"setLed","index":1,"color":"red","r":255,"g":0,"b":0}
```

清除所有 LED：

```json
{"cmd":"clearLeds"}
```

設定所有 LED：

```json
{"cmd":"setAllLeds","color":"blue","r":0,"g":0,"b":255}
```

設定亮度：

```json
{"cmd":"setBrightness","brightness":10}
```

顯示暫存陣列：

```json
{
  "cmd":"showBuffer",
  "leds":[
    {"r":255,"g":0,"b":0},
    {"r":0,"g":0,"b":0}
  ]
}
```

從 MVP-B08-1 起，學生端 LED 編號一律採用 1 起算。

| 學生看到的 LED 編號 | 前端 JSON index | ESP8266 內部 WS2812 index |
|---:|---:|---:|
| 1 | 1 | 0 |
| 2 | 2 | 1 |
| 3 | 3 | 2 |
| 4 | 4 | 3 |
| 5 | 5 | 4 |
| 6 | 6 | 5 |
| 7 | 7 | 6 |
| 8 | 8 | 7 |
| 9 | 9 | 8 |
| 10 | 10 | 9 |
| 11 | 11 | 10 |
| 12 | 12 | 11 |

開發要求：

- Blockly 積木顯示 LED 1～12
- 前端送出 JSON index 1～12
- ESP8266 韌體負責轉換為內部 0～11
- 文件、測試按鈕、範例程式不得再使用 index 0 表示第一顆 LED

## 十六、相容性與限制

### 1. WebSerial 限制

- 需要 HTTPS 或 localhost 環境
- GitHub Pages 可使用 HTTPS
- 使用者需要手動授權序列埠
- iPad 與多數手機瀏覽器不適合作為主要操作環境

### 2. 韌體相容性

同一份 ESP8266 統一韌體應支援：

- Blockly Lab
- Scratch / OSEP Scratch

除非必要，不應任意修改 JSON 協定，以免破壞不同平台相容性。

## 十七、開發原則

### 1. 先穩定協定，再擴充積木

SmartRing JSON 協定應保持穩定，避免每新增一個平台就修改韌體。

### 2. 學生介面優先

學生看到的 LED 編號、積木文字與任務說明應以國中小學生容易理解為優先。

### 3. 教師測試功能保留

平台應保留不透過 Blockly 的教師測試按鈕，方便快速確認硬體與序列通訊是否正常。

### 4. 文件與程式同步更新

每次 MVP 完成後，至少應同步檢查：

- `docs/MVP_LOG.md`
- `docs/PLATFORM_UI_PLAN.md`
- `docs/SMART_RING_SERIAL_PROTOCOL.md`

## 二十二、MVP-B11 需求補充

MVP-B11 的重點是重新整理 SmartRing 積木群的教學定位，避免學生只使用現成特效積木，而是透過 DEMO 觀察後，回到暫存陣列、變數、迴圈與函式完成仿作。

### 1. SmartRing 積木群需求

SmartRing 工具箱應分成三大群組：

1. 連線與狀態 / 偵測 / 基礎控制
2. LED 暫存陣列
3. LED 圖樣、狀態顯示與動畫示範

第一群提供硬體連線、按鈕偵測、等待與直接 LED 控制。第二群是陣列學習核心，學生需先修改暫存陣列，再使用「顯示暫存陣列到 SmartRing」輸出。第三群定位為課程任務 DEMO，執行後直接顯示效果，讓學生觀察後回到第二群積木進行仿作。

### 2. RGB 控制需求

平台應同時保留兩種 RGB 學習層次：

- 單一通道設定：`暫存陣列第 [N] 顆 LED 顏色 [R/G/B] 為 [0~30]`
- 完整 RGB 設定：`設定暫存陣列第 [N] 顆 LED 紅色為 [0~30] 綠色為 [0~30] 藍色為 [0~30]`

單一通道設定用來支援亮度變化、呼吸燈、漸層與變數控制。完整 RGB 設定用來支援函式參數設計與模組化色彩控制。

### 3. DEMO 積木需求

DEMO 積木應採「執行後直接顯示」設計，不再要求學生另外接「顯示暫存陣列到 SmartRing」。B11 先提供：

- 示範圖樣
- 示範狀態顯示

後續動畫 DEMO 應逐步加入，不宜一次放入過多特效，避免弱化陣列與模組化程式的學習目標。

### 4. 舊積木相容需求

B09 / B10 已經建立過的舊積木定義應盡量保留，避免學生或教師先前存下的 `.blockly.xml` 作品載入失敗。若某些舊積木不再符合 B11 工具箱定位，可保留 block 定義與 generator，但不主動放入工具箱。

### 5. 後續版本建議

B11 完成後，後續版本建議依序為：

```text

MVP-B13｜陣列位移動畫 DEMO 版
MVP-B14｜進階視覺效果 DEMO 版
```

## MVP-B11-1 補充需求：積木版面與 RGB 亮度

### 積木版面需求

SmartRing 積木應盡量採用單列輸入顯示，避免 RGB 類積木在工作區中垂直展開過大，影響學生閱讀與操作。

### RGB 數值需求

學生端 RGB 數值維持 0～30，符合 SmartRing 教學亮度與硬體安全需求。

前端傳送到 ESP8266 前，需將學生輸入值換算成 0～255 顏色值：

```text
前端輸出值 = round(學生輸入值 / 30 * 255)
```

此設計可避免學生輸入 30 時，韌體再套用 brightness 限制後亮度變得過暗。

## MVP-B12 補充需求：基礎動畫 DEMO

### 1. 動畫 DEMO 定位

B12 的動畫積木屬於「課程任務 DEMO」。這些積木執行後直接顯示在 SmartRing 上，不需要另外接「顯示暫存陣列到 SmartRing」。

動畫 DEMO 的目的不是讓學生只使用現成動畫，而是讓學生先觀察 LED 變化規律，再使用暫存陣列、迴圈、等待與函式寫出對應程式。

### 2. B12 應提供的基礎動畫

B12 應提供四個基礎動畫 DEMO：

- 播放閃爍動畫 顏色 [顏色] 次數 [N]
- 播放填滿動畫 顏色 [顏色] 速度 [N]
- 播放清除動畫 速度 [N]
- 播放跑馬燈動畫 顏色 [顏色] 速度 [N]

### 3. 教學對應

- 閃爍動畫：對應重複、等待、全部亮、全部滅。
- 填滿動畫：對應索引 1～12、迴圈逐格設定。
- 清除動畫：對應狀態變化與逐格清除。
- 跑馬燈動畫：對應位置變數、單點移動與索引變化。

### 4. 速度限制

動畫速度參數以毫秒為單位。前端應限制速度範圍，避免學生輸入過小數值造成 WebSerial 指令過密。建議速度範圍為 20～2000 毫秒。

### 5. 後續版本

B12 完成後，後續版本建議依序加入：
